The Challenge

A new product with no design foundation, no team, and a hard launch deadline.

The Code Discovery Platform was heading to market with a bold vision: become the go-to search tool for enterprise development teams. The problem was structural — no design department existed, no processes were in place, and the product had no brand, no UI system, and no validated user model.

I stepped in as Lead Product Designer with a single mandate: establish everything needed to ship a credible first release.

Core Insight

The technical product was ready. The design layer — strategy, research, brand, UI — was completely missing. Building it in 6 weeks required sequencing every decision precisely.

Discovery Framework

I audited everything first — then built a shared knowledge architecture.

Before any design work began, I audited every material the team had produced alongside available open-source references. The knowledge was scattered, contradictory, and unsequenced.

I organized it into a structured discovery framework: a visible map of what was known, what was unknown, and what decisions depended on what.

Why This Mattered

Without a shared framework, every team member held a different version of the product in their head. Misalignment this deep creates rework later — I identified it in week one and resolved it before any UI work started.

Deliverables
  • Discovery framework document
  • Product roadmaps (v1 and beyond)
  • Service mapping for core workflows

Personas

I ran a stakeholder workshop that produced 5 validated user archetypes.

Product and marketing teams had different assumptions about who the users were. I ran a structured workshop with stakeholders to surface and reconcile those assumptions, then mapped them against real developer behavior patterns.

The output was dual: UX personas that drove interface decisions, and marketing personas that shaped positioning and messaging.

Deliverables
  • 5 UX personas with behavioral maps
  • Marketing personas for positioning
  • Stakeholder workshop facilitation guide
Use Cases

I mapped 40+ use cases, categorized for cross-functional team handoff.

I identified and captured a comprehensive set of use cases through collaborative sessions with engineering, product, and domain leads. Every case was tied to a specific persona and a measurable task outcome.

The categorized list served as a shared reference that kept design, development, and QA aligned — eliminating the ambiguity that typically causes scope drift mid-sprint.

Deliverables
  • 40+ use cases by persona and workflow
  • Categorized reference list for cross-functional teams
User Stories

I translated use cases into actionable user stories using affinity mapping.

I converted each use case into concise, stakeholder-reviewed user stories and used affinity mapping to group them into logical functional clusters.

The resulting structure gave the development team a clear view of feature scope and dependencies — organized by user need, not by technical convenience.

Deliverables
  • User stories tied to personas and use cases
  • Affinity maps by functional cluster
Initiatives & Functionality

I defined initiative groups and functionality priorities to anchor the v1 scope.

From user stories, I synthesized the product's initiative structure — a set of clearly bounded capability groups that mapped directly to development sprints.

Each group carried prioritization metadata: which persona it served, which use cases it addressed, and its dependency relationships. This eliminated ambiguity when the team had to decide what to cut for the first release.

Deliverables
  • Initiative lists and functional groups
  • Prioritized functionality per release stage
Products Ecosystem

I mapped the full product ecosystem to understand cross-product dependencies before designing anything.

The platform wasn't a single product — it was an ecosystem of interconnected modules. I mapped each subproduct with dedicated mind maps and analyzed how they interacted with each other and with different user types.

This mapping revealed critical UI decisions that would have been impossible to make correctly without understanding the full system — specifically around the Library, Community, and Dependencies modules.

Deliverables
  • Mind maps for all 3 core subproducts
  • Cross-product interaction diagrams
Viable Version & Release Map

I scoped the viable version in detail, then cut a precise release map for v1.

Once the full product vision was mapped, I built a detailed viable version — analyzing user feedback signals, testing assumptions against the ecosystem map, and defining the minimum cohesive feature set for a meaningful launch.

From the viable version, I extracted the release map: a prioritized cut of features confirmed for v1, with rationale. This became the contract between design, product, and engineering.

Deliverables
  • Viable version mind map with prioritization metadata
  • v1 release scope definition

UI Design / Widget

I designed the widget — the most critical user touchpoint — with dark theme and full state coverage.

The widget was the entry point most developers would encounter first — embedded in their IDE or browser context. It required its own dedicated design pass: dark theme by requirement, multiple interaction states, and tight integration with the web platform's data model.

I delivered a complete state matrix (idle, searching, loading, results, empty, error), an interactive prototype, and dev-ready handoff materials.

Design Decision

Dark theme wasn't cosmetic — developer tools that live inside dark IDEs must match the ambient color environment or they create eye strain. This was a functional constraint, not a style preference.

Deliverables
  • Widget UI — dark theme, all states
  • Interactive prototype

Lessons Learned

Three things I'd repeat on every 0→1 product engagement.

Lesson 01
Knowledge alignment is the first design deliverable.
Before a single wireframe, I spent time mapping what the team actually knew — and discovered that different people held different mental models of the same product. Resolving that first saved weeks of rework later.
Lesson 02
Frameworks compound. Research built now pays dividends for all future releases.
The personas, use cases, and functional maps built for v1 didn't expire. They became the reference architecture that every subsequent sprint planned against. Investing time in foundations is never waste — it's infrastructure.
Lesson 03
Dark theme for developer tools is a functional constraint, not aesthetics.
Designing the widget in dark theme from the start — rather than adapting a light theme — produced a far better result. Developer tools live inside dark contexts. Design should start there, not retrofit to it.
